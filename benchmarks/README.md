# Supplier Clustering Engine — Benchmark Pack

A lightweight, repeatable test harness for evaluating clustering quality
across representative input files and collecting structured team feedback.
No LLM calls are required to run benchmarks.

---

## Directory layout

```
benchmarks/
├── README.md                     ← this file
├── input/                        ← benchmark input CSV/XLSX files (one per scenario)
├── expected/                     ← optional expected-output CSV files (same stem as input)
├── outputs/                      ← generated outputs, one sub-folder per input file
│   └── <file_stem>/
│       ├── final_supplier_clustered.csv
│       ├── run_metadata.json
│       └── (other standard outputs)
├── reports/                      ← summary reports
│   └── benchmark_summary.csv     ← generated by run_benchmark_pack.py
└── team_feedback_template.csv    ← copy and fill out to report issues
```

---

## How to add benchmark files

1. Place a CSV or XLSX file in `benchmarks/input/`.  
   Use a descriptive filename, e.g. `danone_sample.csv`, `small_generic_test.csv`.

2. *(Optional)* Place a matching expected-output CSV in `benchmarks/expected/`
   with the **same stem**, e.g. `benchmarks/expected/danone_sample.csv`.
   The expected file must contain `Cluster Number` and `Match Percentage` columns.

3. Run the benchmark pack:
   ```
   python -m scripts.run_benchmark_pack
   ```

4. Review `benchmarks/reports/benchmark_summary.csv` and the per-file output folders.

> **Privacy note:** Do NOT commit real client data into `benchmarks/input/` unless
> it has been explicitly approved for this repository. Use anonymised or synthetic
> test data instead.

---

## How to run the benchmark script

```bash
# Run against all files in benchmarks/input/  (default)
python -m scripts.run_benchmark_pack

# Run against a specific input directory
python -m scripts.run_benchmark_pack --input-dir path/to/files

# Override output and report destinations
python -m scripts.run_benchmark_pack \
    --output-dir benchmarks/outputs \
    --report-dir benchmarks/reports
```

The script does **not** require an OpenAI key. It runs in deterministic
(non-LLM) mode.

## How to use the feedback template

1. Copy `team_feedback_template.csv` (or open it in a spreadsheet application).
2. Fill out one row per issue you want to report.
3. Submit the filled file to the engineering team (e.g. via Slack or email).

The template columns are:

| Column | Purpose |
|---|---|
| `client_file` | Source file name (without path) |
| `supplier_name` | Supplier name from the file |
| `address` | Address from the file |
| `city` | City from the file |
| `country` | Country from the file |
| `cluster_number` | Cluster Number from the output |
| `match_percentage` | Match Percentage from the output (98%, 85%, 70%, blank) |
| `issue_type` | Category of the issue — see list below |
| `expected_result` | What the correct outcome should be |
| `comments` | Free-text notes, context, or additional evidence |
| `reviewer` | Your name or initials |
| `date_reviewed` | Date you reviewed (YYYY-MM-DD) |

---

## Issue type classification

| Issue type | Meaning |
|---|---|
| `Wrong 98` | Two rows are at 98% but should NOT be the same entity |
| `Wrong 85` | Two rows are at 85% but should NOT be in the same family cluster |
| `Wrong 70` | A 70% review candidate is clearly wrong |
| `Missing 98` | Two rows are clearly duplicates but were NOT clustered at 98% |
| `Missing 85` | Two rows are the same supplier family but were NOT clustered at 85% |
| `Missing 70` | A plausible pair was not surfaced for review |
| `Should be blank` | A pair was clustered but should have no relationship |
| `Client/internal domain issue` | An email domain belonging to the client is being used as supplier evidence |
| `Alias needed` | A known brand alias is missing (e.g. Thermo Fisher ↔ Fisher Scientific) |
| `Generic-word issue` | A generic word (equipment, solutions, …) is incorrectly bridging unrelated suppliers |
| `Mapping issue` | The wrong column was auto-detected as supplier name / address / country |
| `LLM needed` | This case needs LLM/manual review and the 70% path was not triggered |

---

## Score meanings

| Score | Meaning |
|---|---|
| **98%** | Confirmed duplicate — same legal entity, same address context |
| **85%** | Supplier family / strong brand match — likely the same parent or known division |
| **70%** | Review candidate — plausible relationship, needs LLM or manual confirmation |
| *(blank)* | No clear relationship detected |
